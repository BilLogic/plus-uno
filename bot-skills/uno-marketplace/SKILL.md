# uno-marketplace

Manage the Plus Prototype Marketplace from Slack. Read, add, edit — never delete (destructive operations belong in a code review, not a Slack message).

## When to Use (which tool to invoke)

The agent invokes **one of three tools** based on the designer's intent:

| User says... | Tool to invoke | Side effect |
|--------------|----------------|-------------|
| "show me toolkit prototypes" | `marketplace_search({ query: 'pillar:toolkit' })` | None — read-only |
| "who built 1008?" | `marketplace_search({ query: 'id:1008' })` | None — read-only |
| "list all high-fi prototypes" | `marketplace_search({ query: 'stage:high' })` | None — read-only |
| "list all prototypes by Bryan" | `marketplace_search({ query: 'creator:Bryan' })` | None — read-only |
| "add my prototype Foo" | `marketplace_add({ ...metadata })` (with confirmation) | Opens PR |
| "register my prototype at /playground/foo" | `marketplace_add({ ...metadata })` (with confirmation) | Opens PR |
| "update 1008's deployment URL to X" | `marketplace_edit({ id: '1008', fields: { deploymentUrl: 'X' } })` (with confirmation) | Opens PR |
| "mark 1011 as high-fi" | `marketplace_edit({ id: '1011', fields: { stage: 'high' } })` (with confirmation) | Opens PR |
| "delete prototype 1009" | **DECLINE.** No delete operation. | — |

**Do NOT use these tools for:**

- Deleting prototypes — not supported in v2. Designer does it manually in `prototypes-data.js` + PR.
- Updating `upvotes` or `lastUpdated` — `upvotes` is managed in-app; `lastUpdated` is auto-set on add/edit.
- Bulk operations (add 5 prototypes at once) — invoke `marketplace_add` once per prototype, not a batch.
- Reading data outside `prototypes-data.js` — for repo-wide questions, route to `uno-qa`.

## Tool: `marketplace_search`

**Read-only, no confirmation gate.** Returns matching entries from `src/pages/PrototypeMarket/prototypes-data.js`.

### Inputs

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `query` | string | Yes | Filter expression. See "Query syntax" below. |
| `limit` | number | No | Default 10, max 25 |

### Query syntax

The agent translates natural language into a filter expression. Supported keys (mirror the schema in `prototypes-data.js`):

- `id:<4-digit-id>` — exact match (e.g. `id:1008`)
- `title:<substring>` — case-insensitive substring (e.g. `title:report`)
- `stage:<low|mid|high>` — exact match
- `pillar:<admin|home|login|profile|toolkit|training|universal>` — exact match
- `creator:<name>` — exact match in `creators` array
- `contributor:<name>` — exact match in `contributors` array (broader than `creator`)

Multiple terms = AND. Free-text not matching any `key:value` pattern falls back to a fuzzy `title` + `description` search.

### Output

For each match, return:

```
{
  id, title, description, stage, pillar,
  creators, deploymentUrl, repoPath, lastUpdated
}
```

Bot composes a Slack mrkdwn list — see Output Format below.

## Tool: `marketplace_add`

**Opens a PR. Requires confirmation gate before firing.**

### Inputs (all per the marketplace schema)

Required:
- `title` — short display name
- `description` — 1-2 sentence summary
- `stage` — `low` | `mid` | `high`
- `pillar` — one of: `admin`, `home`, `login`, `profile`, `toolkit`, `training`, `universal`
- `creators` — array of creator names (at least one)
- `repoPath` — path to prototype dir relative to repo root, must end with `/` (e.g. `playground/my-proto/`)

Optional (default `null`):
- `deploymentUrl` — Netlify deployment URL
- `notionCardUrl`, `notionCardId` — Notion linkage
- `localPath` — route path in root app (e.g. `/1029`)
- `loomVideoUrl` — Loom walkthrough URL

Auto-generated by the tool:
- `id` — next available 4-digit ID (auto-increment from highest current `id`)
- `lastUpdated` — today's date `YYYY-MM-DD`
- `contributors` — defaults to same as `creators` if not provided
- `upvotes` — always `0` for new entries

### Conversational flow

The agent **asks for missing required fields conversationally** before invoking the tool. Example:

```
User: "add my prototype, it's called Tutor Risk Interventions"
Bot:  "Got it. A few more things:
        - What stage? (low / mid / high)
        - Which pillar? (admin / home / toolkit / training / etc.)
        - 1-2 sentence description
        - Path to the prototype directory (e.g. playground/tutor-risk-interventions/)
       Reply in this thread."
User: [provides fields]
Bot:  [posts proposed entry + asks for ✅ confirmation]
User: ✅
Bot:  [invokes marketplace_add → PR opens]
```

### Confirmation gate

Before invoking the tool, the agent posts a proposed entry preview:

```
🆕 *Proposed marketplace entry:*

  id: '1029'  (auto-assigned)
  title: 'Tutor Risk Interventions'
  description: 'Surfaces at-risk tutors via Tutor Coach signals...'
  stage: 'mid'
  pillar: 'admin'
  creators: ['Bryan']
  repoPath: 'playground/tutor-risk-interventions/'
  ...

React with ✅ to open the PR, or reply with corrections.
```

The agent waits for the ✅ reaction (handled by the orchestration layer — see `pipedream-workflow.md` or the chosen platform's equivalent). On ✅, the tool fires.

### Side effect

The tool dispatches a `repository_dispatch` event with `event_type: marketplace-add` to a new GitHub Action (`marketplace-add.yml`, to be built in Week 2). That Action:

1. Checks out `main`
2. Creates branch `bot/marketplace-add-{id}-{date}`
3. Reads `src/pages/PrototypeMarket/prototypes-data.js`, appends the new entry to the `prototypes` array
4. Commits + pushes
5. Opens a draft PR
6. Posts the PR link back to the Slack thread

## Tool: `marketplace_edit`

**Opens a PR. Requires confirmation gate before firing.**

### Inputs

- `id` — required, 4-digit string
- `fields` — required, object with only the keys to change (e.g. `{ deploymentUrl: 'https://...', stage: 'high' }`)

Editable fields:
- `title`, `description`, `stage`, `pillar`
- `deploymentUrl`, `notionCardUrl`, `notionCardId`, `localPath`, `loomVideoUrl`, `repoPath`
- `creators`, `contributors` (full array replace)

NOT editable via tool (would be destructive or auto-managed):
- `id` (immutable)
- `lastUpdated` (auto-updated on edit)
- `upvotes` (managed in-app)

### Confirmation gate

Before invoking, agent posts the proposed diff:

```
✏️ *Proposed edit to prototype 1008 (Monthly Report):*

  - deploymentUrl: null
  + deploymentUrl: 'https://monthly-report.netlify.app'

  - stage: 'mid'
  + stage: 'high'

React with ✅ to open the PR, or reply with corrections.
```

### Side effect

Same pattern as add: dispatches `event_type: marketplace-edit` to `marketplace-edit.yml`. The Action reads the file, surgically modifies the matching entry, commits, opens a draft PR, links back to Slack.

## Output Format

All tool results render in Slack mrkdwn (per `AGENTS.md` → "Slack output formatting").

### `marketplace_search` result

```
🛒 *Marketplace search: "{query}"* ({n} results)

*1008 — Monthly Report* (high, home)
  by Bill • <https://monthly-report.netlify.app|view deployment> • updated 2026-04-22

*1011 — Tutor Performance* (high, admin)
  by Bryan, Ashley • not deployed • updated 2026-04-15

...

Reply for details on any prototype, or ask me to edit one.
```

### `marketplace_add` success

Posted by the GitHub Action back to the thread:

```
✅ *Added prototype 1029 to marketplace*

  <https://github.com/.../pull/XX|Draft PR #XX> ready for review.
  Title: Tutor Risk Interventions
  Stage: mid • Pillar: admin
```

### `marketplace_edit` success

```
✅ *Updated prototype 1008*

  <https://github.com/.../pull/XX|Draft PR #XX> ready for review.
  Changed: deploymentUrl, stage
```

### Error / decline

```
❌ Couldn't {action}: {reason}.
```

E.g. "Couldn't add: pillar 'misc' isn't valid. Use one of: admin, home, login, profile, toolkit, training, universal."

## Skill-Specific Discipline

The shared bot voice from `AGENTS.md` applies. Marketplace-specific rules:

- **Confirmation gate is non-negotiable for add/edit.** Even if the designer's message is unambiguous ("add Foo with stage low and pillar admin and..."), the agent posts the proposed entry and waits for ✅ before opening the PR. The friction is the feature — it prevents accidental PRs.
- **Auto-increment IDs from the live data.** The tool reads `prototypes-data.js` to find the highest current `id` and assigns `id+1`. Never hand-pick an ID.
- **Validate enums at the tool level.** `stage` must be `low|mid|high`. `pillar` must be one of the 7 known values. Reject malformed input before opening a PR — don't ship a broken catalog entry.
- **One operation per tool call.** No batching. If a designer wants to add 5 prototypes, the agent invokes `marketplace_add` 5 times (each with its own confirmation).
- **`repoPath` must end with `/`.** Per the existing schema convention.

## Forbidden

- No delete operation in v2.
- No editing of `id`, `lastUpdated`, or `upvotes` via tool.
- No bulk operations.
- No PRs to `main` without a confirmation gate.

## Edge Cases

- **Designer asks for add but most fields are missing.** Agent asks for the missing required fields conversationally before invoking the tool. Don't invoke with placeholders.
- **Designer asks to add a prototype with an ID that collides with an existing one.** Tool auto-increments; the designer doesn't choose the ID. If they explicitly say "use ID 1029," check if 1029 is taken — if so, surface the conflict and ask if they want the next available.
- **Designer asks to edit a prototype ID that doesn't exist.** Search first; if no match, reply: "I don't see prototype {id} in the marketplace. Did you mean {nearest match}?"
- **Designer reacts to a confirmation prompt with ❌ or asks for changes.** Agent treats it as a "needs corrections" — re-prompts for the fix, then re-asks for confirmation.
- **Search returns >25 results.** Truncate to 25, append "(showing first 25 of {n} — narrow your query)".
- **The `prototypes-data.js` file has parse errors when the GitHub Action tries to edit it.** Action fails, posts error to Slack thread with the parse error message. Designer fixes manually.

## Cost Profile

- `marketplace_search`: ~$0.005 per call. Reads `prototypes-data.js` (~5K tokens), filters in code, formats output. Tiny.
- `marketplace_add`: ~$0.02 per call (Claude composes proposed entry + handles conversational follow-ups). Plus the GitHub Action run cost (~minimal).
- `marketplace_edit`: ~$0.015 per call. Diff composition is cheaper than full add.

Watch via metrics; budget alert if average `add` cost climbs past $0.05 (suggests the conversational follow-ups are getting too long).
